## v3.14 - 离线重载与云端数据同步版 (最新)

在本次更新中，我们引入了 Supabase 云端数据库同步机制，彻底解决了用户在多台设备间同步数据、或者浏览器缓存被清空导致模拟交易与回测数据全部丢失的痛点。

### 1. 接入 Supabase 云端数据库
*   **功能实现**：在 `app.js` 中集成了 Supabase 客户端，使用用户提供的 Supabase Project URL (`https://zhnpmuyryherlzrrfumk.supabase.co`) 和最新的 Publishable key (`sb_publishable_OJar6H5OV0Qka4UuA0VLzA_U4Q5RzXf`) 进行初始化。
*   **双向数据同步流**：
    *   **增量同步 (Upload)**：在执行模拟挂单、开仓、平仓或定时保存（每 5 秒）时，系统会自动在后台将最新的 `portfolio`（账户资产与持仓）和 `backtest`（策略回测状态）打包以 JSONB 格式增量更新（`upsert`）到云端的 `crypto_advisor_state` 表中。
    *   **乐观加载 (Load)**：用户首次打开或刷新页面时，系统会优先以 LocalStorage 进行瞬时渲染（0延迟），同时在后台发起询问拉取云端最新状态，比对后自动合并还原到本地并更新 UI，实现无缝云端重载。
*   **防抖与同步指示器**：我们在网页 Header 的 API 连接状态栏（以及 `live-pulse` 处）整合了同步指示。当数据正在向云端上传时，状态字样会短暂闪烁“正在同步云端数据...”，同步成功后显示“云端数据已同步”，发生网络断开或域阻断时会自动回退到本地缓存并提示“云端同步失败 (已用本地缓存)”，保证高可用性。

## v3.13 - 离线重载与视图状态持久化版


在本次更新中，我们解决了每次刷新页面或重新打开标签页时，系统默认恢复为“用户手动交易”且币种重置为“BTC”，导致用户看不到策略账户（如策略 C）运行状态和成交历史的 UX（用户体验）缺陷。

### 1. 本地缓存用户最后选中的币种（BTC/ETH）与活跃账户（手动/A/B/C/D）
*   **漏洞定位**：由于 `activeAccountId` 和 `activeToken` 变量仅存在于浏览器的运行内存中，当用户关闭网页再重新打开时，系统会默认以 `manual`（手动账户）和 `BTC`（比特币）启动。如果用户之前运行的活跃订单都在策略账户（如策略 C）下，在默认的手动界面下会显示为“空仓且无交易历史”，这极易让用户产生“昨天的历史成交记录全部丢失了”的错觉。
*   **解决方案**：我们在 `switchToken` 和 `switchActiveAccount` 函数中引入了 LocalStorage 持久化存储。当用户每一次切换币种或切换查看的策略账户时，最新的视图状态就会自动写入本地缓存。
    在网页 DOM 加载完成时（`DOMContentLoaded`），系统会自动读取这两个缓存状态并还原视图，确保用户重新打开页面时**依然停留在上次关闭时的界面**，彻底消除了界面错位产生的虚假“丢失”感知。

## v3.12 - 离线重载与突破挂单逻辑优化版

在本次更新中，我们解决了 AI 回测策略中右侧突破（方案 B）和右侧破位（方案 D）挂单因频繁跟随价格变动导致“永远无法成交”的逻辑缺陷。

### 1. 锁定右侧突破/破位挂单的入场点位
*   **漏洞定位**：方案 B（右侧突破）和方案 D（右侧破位）的挂单触发价是基于当前最新价（`currentPrice`）加上或减去固定百分比计算得来的。
    由于系统每 5 秒就会自动运行一次计算并校准策略，原先的代码会将已创建的未成交挂单按照新的最新价反复重置。这就导致挂单价格始终在跟着最新价“等距奔跑”（价格涨，挂单价也涨；价格跌，挂单价也跌），除非出现超大波动直接穿透，否则在普通行情中**几乎永远不可能被成交**。
*   **解决方案**：我们修改了 `syncBacktestOrders` 中的更新条件。现在，只要方案 B 或方案 D 的挂单已经创建，系统将**强行锁定其初始挂单价**，停止跟随价格闪烁。只有在用户手动切换交易币种（BTC/ETH）时才会重置。这使突破和破位挂单可以像真实的交易单一样固定在特定点位静候价格突破，大大提高了回测数据的撮合效率和拟真度。

## v3.11 - 离线重载与纯净控制台优化版

在本次更新中，我们解决了控制台由于缺少网站图标导致的 `GET /favicon.ico 404 (Not Found)` 报错，达成了 100% 纯净无红字错误的控制台体验。

### 1. 引入 Base64 格式透明占位 Icon 阻断默认请求
*   **漏洞定位**：现代浏览器在渲染 HTML 页面时，会自动向服务器的根路径请求 `/favicon.ico` 文件以显示在标签页上。由于本地 HTTP 服务器（`http-server`）只提供项目文件，在缺少此文件的直连部署下，浏览器请求会被服务器拦截并抛出 `404 Not Found`。
*   **解决方案**：我们在 `index.html` 的 `<head>` 中嵌入了标准的 Base64 透明像素图标声明：
    `<link rel="icon" href="data:;base64,iVBORw0KGgoAAAANSUhEUgAAAAEAAAABCAYAAAAfFcSJAAAADUlEQVR42mNk+M9QDwADhgGAWjR9awAAAABJRU5ErkJggg==">`
    这在不产生额外 HTTP 请求负荷的情况下，使浏览器直接通过内置的透明像素解析占位图，完全消除了控制台的 `/favicon.ico 404` 错误。

## v3.10 - 离线重载与OKX/CORS兼容防拦截优化版

在本次更新中，我们解决了某些特定地区因交易所主站 IP 物理风控拦截（HTTP 451 / Legal Reasons）导致的浏览器 CORS 报错及卡顿。

### 1. 调整备用 API 调用优先级，默认屏蔽/推迟易遭物理封锁的接口
*   **漏洞定位**：在之前的版本中，部分技术指标和工具函数（如资金费率、均线评估、24小时涨跌幅、美元/离岸人民币汇率等）首选调用的是币安官方主 API (`api.binance.com` / `fapi.binance.com`)。
    由于部分国内用户的网络环境或未开启 VPN 代理的直连环境，会导致币安接口返回 `HTTP 451 (Unavailable For Legal Reasons)` 并剥离 CORS 头。由于该接口是第一优先级，浏览器在每次定时更新时都会尝试请求，在控制台抛出大量的红色 CORS 跨域风控报错。
*   **解决方案**：我们重新对各个核心更新指标的 API 请求链路进行了重排：
    1. **资金费率、均线、4H趋势、挂单深度**：全面将支持 CORS 且国内直连无封锁的 **OKX V5 API** 调整为第一优先级，将币安主 API 作为最后的非跨域兜底降到最低优先级。
    2. **美元/离岸人民币汇率 (USDTCNH)**：将支持跨域且全球直连的 **OpenER 汇率 API** 调整为第一优先级，彻底规避因直连币安外汇接口导致的 451 风控拦截。
    *   **最终效果**：在完全不挂载代理 VPN 的国内浏览器运行环境下，页面首屏加载及定时轮询不再尝试优先握手易受阻的 `api.binance.com`，从而在根本上阻断了浏览器控制台产生 451 / CORS 跨域风控报错的可能性。

## v3.9 - 离线重载与OKX/CORS兼容优化版

在本次更新中，我们彻底解决了由 Gate.io 接口不支持浏览器跨域（CORS）所导致的控制台频繁报错、以及无法成功执行备用数据调用的问题，并按照用户要求引入了 OKX 数据的直接调用。

### 1. 彻底用 OKX V5 和 HTX V1 替换 Gate.io 作为备用/兜底 API
*   **漏洞定位**：原先的 Gate.io API 接口 (`api.gateio.ws`) 服务器未返回 `Access-Control-Allow-Origin` 响应头，这就导致当币安主接口被 GFW 拦截或网络超时，系统尝试向 Gate.io 请求价格、资金费率、EMA 均线和深度数据时，会被浏览器 CORS 策略拦截，在控制台抛出大量的 `TypeError: Failed to fetch`。
*   **解决方案**：我们用完全支持 CORS 跨域请求的 **OKX V5 API** 替换了所有 Gate.io 的数据源，实现了对 OKX 数据的高效直接调用。同时，为防万一，我们还将完全支持 CORS 的 **HTX (火币) API** 接入了价格查询的最底层作为第四级兜底。这不仅完美解决了浏览器中的 CORS 跨域拦截错误，还为国内无代理的网络环境下提供了极高成功率的备用数据源。

## v3.8 - 离线重载与重置竞态锁修复版

在本次更新中，我们彻底排查并修复了导致用户在离线数小时后，重新打开网页时出现**“挂单与当前持仓全部意外消失且没有撮合交易历史”**的关键竞态条件缺陷。

### 1. 独立重置计数器 (resetCounter) 隔离定时器干扰
*   **漏洞定位**：在先前的设计中，为了防止用户在下载历史 K 线期间点击“重置账户”而引发数据覆盖，我们在离线同步引擎中加入了解析前后的时间戳校验：`if (portfolio.lastSyncTime !== startSyncTime) return;`。
    然而，系统内置的 **5秒定时自动保存器** 在网页刚加载并下载 K 线期间（通常需要 2-4 秒）也会自动更新 `portfolio.lastSyncTime = Date.now()`。这就导致离线引擎下载完 K 线时，误以为用户执行了“账户重置”，进而**强行终止了历史撮合结果的写入与保存**。
*   **后果**：由于写入被强行中止，系统无法在追溯结束时调用 `runCalculator()`，导致所有 AI 策略均处于挂单和持仓均为空的“植物人状态”。
*   **解决方案**：我们引入了一个独立的全局计数器 `resetCounter`。该计数器**仅在用户明确点击“重置账户”或系统底层抛出异常需要强制重算时**才会累加。离线同步引擎现在只校验 `resetCounter` 是否发生变化。这完美避开了 5 秒定时器的常规状态保存干扰，确保离线追溯与撮合 100% 写入成功。

## v3.7 - 离线时间校准版

### 1. 规避本地系统时钟与交易所的时间线差值
*   **漏洞定位**：由于您当前电脑的系统时间处于未来的 **2026年**，而币安/Gate.io 交易所的物理服务器运行在现实时间的 UTC 时间线上。当我们向交易所请求 2026 年的时间戳时，交易所服务器会因“请求未来时间”而返回空数据。
*   **解决方案**：我们重新设计了 `fetchHistoricalKlines`。去除了 `startTime` 和 `endTime` 参数，改用根据离线时长自动换算出的 K 线根数（`limit`）向交易所直接请求**最近的 X 根 K 线**。获取数据后，代码会自动计算其与您系统时间的时差（约 2 年），并将所有的历史 K 线时间戳整体平移投影到 2026 年，实现跨时空无感重放。
